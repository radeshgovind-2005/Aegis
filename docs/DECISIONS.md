## 2026-04-22 — Defer coverage instrumentation to Phase 2 via split test environments

**Context:** CI on PR #2 failed because `vitest run --coverage` tried to instrument via `@vitest/coverage-v8`, which internally uses `node:inspector.Session`. The `@cloudflare/vitest-pool-workers` pool executes tests inside workerd, where `node:inspector` is not implemented. The two packages are structurally incompatible in a single vitest config.

**Options:**
- A: Custom coverage provider module shim. *Rejected — exotic, poorly documented, high maintenance.*
- B: Split configs now (one without the workers pool for unit tests, one with for adapter/integration). Architecturally correct but out of scope for task 0.1, whose remit is "wire lint/typecheck/test/verify", not "design the test topology".
- C: Drop `--coverage` from Phase 0 CI, uninstall `@vitest/coverage-v8`, and schedule the split as the first task of Phase 2. **Chosen.**

**Decision:** Remove `--coverage` from CI. Uninstall `@vitest/coverage-v8`. Add task 2.0 to Phase 2 covering the split and coverage re-enable.

**Rationale:** Phase 0 has no domain code, so there is nothing to measure coverage of. `check-coverage.mjs` already has a "no summary found, skipping" fallback — the gate is latent, not broken. Phase 2 is when domain code appears and coverage becomes meaningful; it is also when the test-environment split is unavoidable regardless. Do the split where it has a natural home.

**Consequences:** Phase 0 CI has no coverage measurement. Intentional. Phase 2 must complete task 2.0 before any domain task is started, or the coverage gate will not be honored.

**Links:** PR #2 CI failure logs, docs/phases/phase-02-domain-core.md task 2.0

## 2026-04-22 — Sabotage the `deploy` script; pin deploys to GitHub Actions

**Context:** The `deploy` script generated by `wrangler init` runs `wrangler deploy` directly. Any developer (or Claude Code session) could accidentally deploy to production by running `npm run deploy`. Phase 0 locks this down immediately.

**Options:**
- Leave as-is and rely on convention. *Rejected — too easy to bypass by mistake.*
- Remove the `deploy` key entirely. *Rejected — CI may reference the script name.*
- Sabotage the script to print a message and exit 1. **Chosen.**

**Decision:** `"deploy": "echo 'Refusing: use GitHub Actions workflow_dispatch for production deploys.' && exit 1"`

**Rationale:** Makes the guard visible, self-documenting, and resistant to muscle memory. Any future relaxation requires an explicit code change and a new DECISIONS.md entry explaining why.

**Consequences:** Running `npm run deploy` locally always fails. Intentional. CI deploy goes through `workflow_dispatch` in Phase 8.

**Links:** docs/_package-json-additions.md, docs/phases/phase-00-scaffold.md task 0.1, Phase: 00 | Task: 0.1-package-scripts

## 2026-04-22 — Pin ESLint to v8; defer v9 flat-config migration

**Context:** The spec doc (`docs/_package-json-additions.md`) calls for `eslint@^9`. The existing `.eslintrc.cjs` uses the legacy config format (v8 style). ESLint v9 deprecated legacy config and `eslint-plugin-import@^2` has known compatibility gaps with v9's legacy-compat layer.

**Options:**
- Install `eslint@^9` and use legacy compat shim. *Risky — import plugin compat unstable.*
- Install `eslint@^9` and rewrite to flat config now. *Scope creep for a tooling-only task.*
- Pin `eslint@^8`; migrate to flat config later. **Chosen.**

**Decision:** `eslint@^8.x` with the existing `.eslintrc.cjs`. A TODO entry tracks the migration.

**Rationale:** Getting `npm run verify` to exit 0 cleanly is the task's goal. Introducing a known unstable compat layer risks breaking CI before the project even has a single test. The migration is a mechanical rename, not a design decision — defer it.

**Consequences:** Using a deprecated ESLint major. Acceptable for a scaffold phase. `docs/_package-json-additions.md` references `eslint@^9` but the phase file is authoritative; the spec doc should be updated to match (tracked in TODO).

**Links:** docs/phases/phase-00-scaffold.md task 0.1, docs/TODO.md, Phase: 00 | Task: 0.1-package-scripts

## 2026-04-22 — Add tsconfig.eslint.json to widen ESLint's TS project scope

**Context:** The main `tsconfig.json` excludes `test/` and does not include `vitest.config.mts`. The `@typescript-eslint/parser` requires every file it lints to be in the TS project specified by `parserOptions.project`. Without a separate config, ESLint crashes on those files with "file was not found in any of the provided project(s)".

**Options:**
- Add `test/` and `vitest.config.mts` to the main `tsconfig.json`. *Rejected — the main tsconfig intentionally excludes test files to keep the production build surface clean.*
- Set `parserOptions.createDefaultProgram: true`. *Rejected — deprecated and slow.*
- Create `tsconfig.eslint.json` extending main tsconfig with broader includes. **Chosen.**

**Decision:** `tsconfig.eslint.json` extends `./tsconfig.json`, overrides `include` to add `test/**/*.ts`, `vitest.config.mts`, `.eslintrc.cjs`, and clears `exclude`. `.eslintrc.cjs` `parserOptions.project` points at the new file.

**Rationale:** Standard pattern for TypeScript + ESLint repos. Keeps main tsconfig clean while giving the linter full visibility. The eslint config file itself is included so the linter can lint the linter config (useful for catching syntax errors in `.eslintrc.cjs`).

**Consequences:** Two tsconfigs to maintain. Low burden — `tsconfig.eslint.json` is a thin extension and rarely needs touching.

**Links:** tsconfig.eslint.json, .eslintrc.cjs, Phase: 00 | Task: 0.1-package-scripts

## 2026-04-22 — Boundary guard as tsx script, not vitest test

**Context:** Task 0.2 calls for a guard test proving that the `no-restricted-imports` ESLint rule catches forbidden domain imports. The obvious implementation is a vitest test using ESLint's Node API. However, `vitest.config.mts` uses `@cloudflare/vitest-pool-workers`, which runs all tests inside Miniflare/workerd — an environment where ESLint's complex Node.js dependency tree cannot run. Introducing a second vitest config to support a Node pool was deferred to Phase 2, task 2.0, per DECISIONS.md (2026-04-22 — Defer coverage instrumentation).

**Options:**
- Pre-empt task 2.0: add a `vitest.node.config.mts` now. *Rejected — scope creep; task 2.0 owns this.*
- Vitest test using ESLint API: blocked by Workers pool constraint. *Rejected — technically infeasible without the split.*
- tsx script run by `npm run test:boundary`, included in `npm run verify`. **Chosen.**

**Decision:** `test/domain/boundary.guard.ts` is a plain TypeScript script executed by `tsx` (already installed). It loads the project's `.eslintrc.cjs`, lints an inline import string against a virtual `src/domain/` filepath, and exits non-zero if the violation is not reported. Added `test:boundary` to package.json scripts and appended it to `verify`.

**Rationale:** This is regression coverage for the boundary rule, not a TDD cycle — the rule pre-existed from task 0.1. The tsx script runs in Node.js proper, avoiding the Workers pool issue entirely. The inline-text + virtual-filepath approach (`eslint.lintText`) removes any need to exclude fixture files from the CLI, tsconfig, or vitest. The script is part of `verify`, so CI catches regressions automatically.

**Consequences:** The guard is not a vitest test (no vitest reporter integration, no coverage tracking). This is intentional for Phase 0. When task 2.0 introduces the unit config, the guard could be converted to a vitest test if desired — tracked in TODO.md.

**Links:** test/domain/boundary.guard.ts, package.json scripts.test:boundary, Phase: 00 | Task: 0.2-directory-skeleton

## 2026-04-21 — Repository scaffold and guardrails

**Context:** Project start. Need a structure that lets a fresh Claude Code session pick up deterministically and enforces TDD + hexagonal boundaries + human-in-the-loop reviews.

**Options:**
- Loose structure, rely on prompting per session. *Rejected — drift within one session, catastrophic across sessions.*
- Heavyweight full-DDD project with aggregates, domain events, repositories. *Rejected — scope mismatch; domain is small.*
- Lightweight hexagonal: pure domain, ports, adapters, thin interfaces layer. **Chosen.**

**Decision:** Hexagonal with pure domain and four ports (embedding, vector-index, audit-log, cache). ESLint enforces the domain → no-Cloudflare-imports boundary. Phased roadmap with explicit human checkpoints. Append-only docs as cross-session memory.

**Rationale:** Small domain + infrastructure swap-ability + engineering credibility. The port boundary is what lets us demonstrate the classification logic is pure, which is the whole point of the pitch.

**Consequences:** Slight overhead for Phase 2 — a few extra files for what could be inlined. Worth it because the pitch story ("the domain doesn't know what runs it") is concrete, not hand-waved.

**Links:** CLAUDE.md, docs/ROADMAP.md, docs/adr/0001-hexagonal-architecture.md
